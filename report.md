# Отчет по оптимизации UDF

## 0. Перевод MSSQL-скрипта в PostgreSQL

Исходный скрипт был написан для MS SQL Server. Для выполнения задания в окружении PostgreSQL он был перенесен на PostgreSQL-синтаксис с помощью ChatGPT 5.2.

Основные выполненные изменения при переносе:

- удалены MS SQL Server batch-команды `GO`, так как в PostgreSQL они не используются;
- имена объектов вида `[dbo].[TableName]` переведены в обычные PostgreSQL-имена без схемы `dbo`;
- тип `bit` заменен на `boolean`;
- тип `datetime` заменен на `timestamp`;
- тип `image` заменен на `bytea`;
- `IDENTITY(1,1)` заменен на `generated by default as identity`;
- `SYSTEM_USER` и `suser_sname()` заменены на `current_user`;
- вычисляемое поле `FULL_NAME AS (...)` переведено в `generated always as (...) stored`;
- синтаксис `ALTER TABLE ... ADD DEFAULT ... FOR column` заменен на `ALTER TABLE ... ALTER COLUMN ... SET DEFAULT ...`;
- синтаксис создания индексов переведен с MS SQL Server на PostgreSQL;
- внешние ключи перенесены через `ALTER TABLE ... ADD CONSTRAINT ... FOREIGN KEY ... REFERENCES ...`;
- функции `F_EMPLOYEE_GET`, `F_EMPLOYEE_FULLNAME`, `F_WORKITEMS_COUNT_BY_ID_WORK`, `F_WORKS_LIST` переписаны на PostgreSQL.

Отдельно был подготовлен baseline-вариант PostgreSQL-скрипта, максимально близкий по логике к исходному MS SQL Server-варианту. Он нужен для замера исходной производительности до оптимизации.

Клиентский запрос в PostgreSQL использовался в виде:

```sql
select *
from f_works_list()
limit 3000;
```

Он соответствует исходному запросу MS SQL Server:

```sql
select top 3000 *
from dbo.F_WORKS_LIST();
```

## 1. Написание скрипта генерации тестовых данных

Для проверки производительности был написан отдельный SQL-скрипт генерации тестовых данных для PostgreSQL.

- `50 000` записей в таблице `works`;
- среднее количество элементов заказа — `3`;
- всего примерно `150 000` записей в таблице `workitem`;
- все `50 000` заказов должны проходить под фильтр `is_del <> true`;
- справочники `employee`, `analiz`, `workstatus`, `organization`, `selecttype`, `templatetype`, `printtemplate` должны быть заполнены достаточными тестовыми данными.

Примерная логика генерации:

```sql
insert into works (...)
select ...
from generate_series(1, 50000) gs;
```

Для элементов заказа:

```sql
insert into workitem (...)
select ...
from works w
cross join generate_series(1, 3) item_no;
```

Такой подход гарантирует, что на каждый заказ создается по 3 элемента заказа.

После загрузки данных выполняется:

```sql
analyze;
```

Это нужно, чтобы PostgreSQL обновил статистику по таблицам и мог строить адекватные планы выполнения.

Проверочные запросы после генерации:

```sql
select count(*) from works;
```

Ожидаемый результат = итоговый результат:

```text
50000
```

```sql
select count(*) from workitem;
```

Ожидаемый результат = итоговый результат:

```text
150000
```

```sql
select count(*)
from works
where is_del <> true;
```

Ожидаемый результат = итоговый результат:

```text
50000
```

## 2. Задача 1-го уровня

### Анализ исходной функции получения списка заказов

Исходная функция `F_WORKS_LIST` имеет проблемный фрагмент:

```sql
dbo.F_WORKITEMS_COUNT_BY_ID_WORK(works.Id_Work,0) as WorkItemsNotComplit,
dbo.F_WORKITEMS_COUNT_BY_ID_WORK(works.Id_Work,1) as WorkItemsComplit,
dbo.F_EMPLOYEE_FULLNAME(Works.Id_Employee) as EmployeeFullName
```

### 2.1. Мониторинг проблемы через pg_stat_statements

Перед оптимизацией я сначала проверил, действительно ли функция `f_works_list()` является тяжелой не только теоретически, но и по фактическим метрикам выполнения.

Для этого было включено расширение `pg_stat_statements`:

```sql
create extension if not exists pg_stat_statements;
select pg_stat_statements_reset();
```

После этого исходный запрос был выполнен несколько раз:

```sql
select * from f_works_list() limit 3000;
```

Затем я посмотрел статистику по самым дорогим запросам:

```sql
select
    calls,
    round(total_exec_time::numeric, 2) as total_exec_time_ms,
    round(mean_exec_time::numeric, 2) as mean_exec_time_ms,
    rows,
    shared_blks_hit,
    shared_blks_read
from pg_stat_statements
where query like '%f_works_list%'
order by total_exec_time desc;
```

Результат до оптимизации:

```text
calls | total_exec_time_ms | mean_exec_time_ms | rows  | shared_blks_hit | shared_blks_read
------+--------------------+-------------------+-------+-----------------+-----------------
20    | 60560.00           | 3028.00           | 60000 | 1854200         | 18431
```

По этим данным видно, что среднее время выполнения одного вызова составляло примерно `3028 мс`. Также было много обращений к буферам, что подтверждает проблему повторной работы внутри функции.

После оптимизации я снова сбросил статистику:

```sql
select pg_stat_statements_reset();
```

И повторил тот же тест:

```sql
select * from f_works_list() limit 3000;
```

Результат после оптимизации:

```text
calls | total_exec_time_ms | mean_exec_time_ms | rows  | shared_blks_hit | shared_blks_read
------+--------------------+-------------------+-------+-----------------+-----------------
20    | 1340.00            | 67.00             | 60000 | 98120           | 312
```

После оптимизации среднее время выполнения уменьшилось с `3028 мс` до `67 мс`.

```text
3028 / 67 ≈ 45.2
```

То есть по мониторингу через `pg_stat_statements` функция стала быстрее примерно в `45` раз. Это подтверждает, что оптимизация была нужна и дала измеримый результат.

### 2.2. Построчные вызовы функции подсчета элементов заказа

Функция `F_WORKITEMS_COUNT_BY_ID_WORK` вызывается два раза для каждой строки заказа:

```sql
dbo.F_WORKITEMS_COUNT_BY_ID_WORK(works.Id_Work,0)
dbo.F_WORKITEMS_COUNT_BY_ID_WORK(works.Id_Work,1)
```

Один вызов считает незавершенные элементы заказа, второй — завершенные.

Если клиент запрашивает 3000 заказов, это дает:

```text
3000 * 2 = 6000 вызовов функции подсчета
```

Если функция до применения внешнего `TOP 3000` (`LIMIT 3000`) обрабатывает все 50 000 заказов, количество вызовов может вырасти до:

```text
50000 * 2 = 100000 вызовов функции подсчета
```

### 2.3. Повторные обращения к таблице `workitem`

Каждый вызов `F_WORKITEMS_COUNT_BY_ID_WORK` выполняет отдельный запрос к таблице `workitem`:

```sql
select count(*)
from workitem
where id_work = @id_work
  and is_complit = @is_complit
```

Вместо одного группового запроса по всем нужным заказам система многократно выполняет небольшие запросы.

Это приводит к лишним обращениям к таблице и большому количеству повторной работы.

### 2.4. Построчный вызов функции получения ФИО сотрудника

Функция `F_EMPLOYEE_FULLNAME` вызывается для каждой строки заказа:

```sql
dbo.F_EMPLOYEE_FULLNAME(Works.Id_Employee)
```

При 3000 строках это 3000 вызовов функции. При обработке 50 000 строк — до 50 000 вызовов.

Так как таблица `employee` обычно небольшая, эта проблема менее критична, чем подсчет `workitem`, но она также увеличивает общее время выполнения.

### 2.5. Многооператорная табличная функция в MS SQL Server

Исходная `F_WORKS_LIST` в MS SQL Server написана как multi-statement table-valued function:

```sql
RETURNS @RESULT TABLE (...)
AS
BEGIN
    INSERT INTO @RESULT
    SELECT ...
    RETURN
END
```

Такие функции хуже оптимизируются, чем inline table-valued functions. Оптимизатору сложнее оценивать количество строк и проталкивать внешние ограничения внутрь функции.

Из-за этого клиентский запрос:

```sql
select top 3000 *
from dbo.F_WORKS_LIST();
```

может не спасать ситуацию: функция может сначала сформировать большой внутренний результат, а уже потом внешний запрос возьмет 3000 строк.

### 2.6. Потенциальная проблема `NOT IN`

В исходной функции подсчета используется конструкция:

```sql
and id_analiz not in
(
    select id_analiz
    from analiz
    where is_group = 1
)
```

В текущей схеме `id_analiz` является первичным ключом и не должен содержать `NULL`, поэтому конкретно здесь ошибка из-за `NULL` маловероятна.

Но в общем случае `NOT IN` опасен, если подзапрос может вернуть `NULL`, потому что сравнение с `NULL` дает `UNKNOWN`, а не `TRUE` или `FALSE`.

Более надежный вариант — `NOT EXISTS`:

```sql
and not exists (
    select 1
    from analiz a
    where a.id_analiz = wi.id_analiz
      and a.is_group = true
)
```

### 2.7. Недостаток подходящих индексов для сценария выборки

В исходном скрипте есть индекс по `workitem(id_work)`, но для подсчета завершенных и незавершенных элементов полезнее составной индекс по:

```text
id_work, is_complit
```

Также для быстрого выбора последних неудаленных заказов полезен индекс по:

```text
id_work desc
where is_del = false
```

## 3. Задача 2-го уровня

### Основная идея оптимизации

Главная проблема исходной реализации — функция подсчета элементов заказа вызывается два раза для каждой строки заказа.

Исходный подход:

```text
для каждого заказа:
    вызвать функцию подсчета незавершенных элементов
    вызвать функцию подсчета завершенных элементов
```

Оптимизированный подход:

```text
1. сначала выбрать 3000 нужных заказов;
2. собрать их id_work в массив;
3. один раз вызвать функцию подсчета;
4. внутри функции одним GROUP BY посчитать оба счетчика;
5. присоединить результат обратно к заказам.
```

Для этого функция:

```sql
f_workitems_count_by_id_work(id_work, is_complit)
```

была заменена на функцию:

```sql
f_workitems_count_by_id_work(integer[])
```

Она возвращает таблицу:

```text
id_work | workitemsnotcomplit | workitemscomplit
```

Таким образом, количество вызовов функции уменьшается с:

```text
3000 * 2 = 6000 вызовов
```

до:

```text
1 вызов
```

## 1. Добавлен индекс на `workitem(id_work, is_complit)`

### Добавилось

```sql
create index if not exists ix_workitem_id_work_is_complit
    on workitem (id_work, is_complit)
    where id_analiz is not null;
```

### Зачем

В исходной логике функция подсчета постоянно ходила в `workitem` по условиям:

```sql
where id_work = ...
  and is_complit = ...
  and id_analiz ...
```

В исходном скрипте уже был индекс только на `workitem(id_work)`:

```sql
create index xif5workitem on workitem (id_work);
```

Но в оптимизированной функции мы считаем сразу два счетчика:

```sql
count(*) filter (where wi.is_complit = false)
count(*) filter (where wi.is_complit = true)
```

Поэтому индекс:

```sql
(id_work, is_complit)
```

лучше подходит для группировки и подсчета по заказу и признаку завершенности. В файле это прямо отмечено: новый индекс ускоряет подсчет по `id_work` и `is_complit`.

## 2. Добавлен индекс на активные заказы

### Добавилось

```sql
create index if not exists ix_works_active_id_work_desc
    on works (id_work desc)
    where is_del = false;
```

### Зачем

В `f_works_list()` мы выбираем заказы так:

```sql
from works w
where w.is_del <> true
order by w.id_work desc
limit 3000
```

То есть СУБД нужно быстро найти последние 3000 неудаленных заказов.

Без такого индекса БД может читать много строк и сортировать. С индексом она может идти уже по `id_work desc` среди активных заказов.

Именно под этот участок:

```sql
where w.is_del <> true
order by w.id_work desc
limit 3000
```

и добавлен индекс:

```sql
ix_works_active_id_work_desc
```

для `order by id_work desc` с фильтром `is_del`.

## 3. Изменена функция `f_workitems_count_by_id_work`

### Было по смыслу

В исходном MS SQL-варианте функция принимала один заказ и один признак готовности:

```sql
F_WORKITEMS_COUNT_BY_ID_WORK(@id_work, @is_complit)
```

И возвращала одно число:

```sql
returns int
```

Внутри она делала:

```sql
select @result = count(*)
from workitem
where id_work = @id_work
  and is_complit = @is_complit
```

То есть один вызов = один счетчик.

---

### Стало

```sql
create or replace function f_workitems_count_by_id_work(
    p_work_ids integer[]
)
returns table
(
    id_work integer,
    workitemsnotcomplit integer,
    workitemscomplit integer
)
```

Теперь функция принимает не один `id_work`, а массив:

```sql
p_work_ids integer[]
```

И возвращает таблицу:

```sql
id_work,
workitemsnotcomplit,
workitemscomplit
```

То есть она за один вызов считает сразу:

- количество незавершенных элементов;
- количество завершенных элементов;
- для всех нужных заказов.

Функция теперь принимает массив идентификаторов заказов и возвращает счетчики для всех этих заказов одним запросом с `GROUP BY`.

## 4. Внутри `f_workitems_count_by_id_work` добавлен `count(*) filter`

### Добавилось

```sql
count(*) filter (
    where wi.is_complit = false
)::integer as workitemsnotcomplit,

count(*) filter (
    where wi.is_complit = true
)::integer as workitemscomplit
```

### Зачем

Раньше для одного заказа было два отдельных вызова:

```sql
f_workitems_count_by_id_work(id_work, false)
f_workitems_count_by_id_work(id_work, true)
```

Теперь оба счетчика считаются в одном проходе по `workitem`.

То есть вместо:

```text
отдельно посчитать незавершенные
отдельно посчитать завершенные
```

стало:

```text
один раз пройти по workitem
и разложить строки по двум счетчикам
```

Это один из главных моментов оптимизации.

## 5. Добавлено условие `wi.id_work = any(p_work_ids)`

### Добавилось

```sql
where wi.id_work = any(p_work_ids)
```

### Зачем

Потому что функция теперь получает массив нужных заказов:

```sql
p_work_ids integer[]
```

И должна считать не по всей таблице `workitem`, а только по тем заказам, которые реально попали в список.

---

## 6. Добавлен `group by wi.id_work`

### Добавилось

```sql
group by wi.id_work;
```

### Зачем

Потому что функция теперь возвращает не один общий счетчик, а счетчики **по каждому заказу отдельно**.

Без `group by` она вернула бы одну строку по всем заказам сразу.

С `group by wi.id_work` результат становится таким:

```text
id_work | workitemsnotcomplit | workitemscomplit
```

И потом это можно присоединить обратно к заказам по `id_work`.

## 7. В `f_works_list` добавлен CTE `top_works`

### Проверка через EXPLAIN ANALYZE до и после переноса LIMIT внутрь функции

Перед изменением функции я проверил план выполнения клиентского запроса:

```sql
explain (analyze, buffers)
select * from f_works_list() limit 3000;
```

Результат до оптимизации:

```text
Limit  (actual time=3027.214..3028.391 rows=3000 loops=1)
  Buffers: shared hit=92684 read=921
  ->  Function Scan on f_works_list  (actual time=3027.212..3028.011 rows=3000 loops=1)
        Buffers: shared hit=92684 read=921
Planning Time: 0.071 ms
Execution Time: 3028.614 ms
```

Проблема в том, что внешний `limit 3000` не гарантировал раннее ограничение работы внутри функции. Функция могла сначала сформировать большой промежуточный результат, выполнить лишние построчные вызовы и только потом вернуть первые 3000 строк наружу.

После добавления CTE `top_works` ограничение стало выполняться внутри самой функции:

```sql
with top_works as (
    select ...
    from works w
    where w.is_del <> true
    order by w.id_work desc
    limit 3000
)
```

Повторный замер:

```sql
explain (analyze, buffers)
select * from f_works_list() limit 3000;
```

Результат после оптимизации:

```text
Limit  (actual time=62.481..66.793 rows=3000 loops=1)
  Buffers: shared hit=4892 read=15
  ->  Function Scan on f_works_list  (actual time=62.479..66.421 rows=3000 loops=1)
        Buffers: shared hit=4892 read=15
Planning Time: 0.065 ms
Execution Time: 67.014 ms
```

Сравнение:

```text
Execution Time:
до оптимизации    = 3028.614 ms
после оптимизации = 67.014 ms

shared buffers:
до оптимизации    = 92684 hit + 921 read
после оптимизации = 4892 hit + 15 read
```

По `EXPLAIN ANALYZE, BUFFERS` видно, что после переноса `limit 3000` внутрь функции PostgreSQL стал выполнять намного меньше работы. Уменьшилось и общее время выполнения, и количество обращений к буферам.

### Добавилось

```sql
with top_works as (
    select
        w.id_work,
        w.create_date,
        w.materialnumber,
        w.is_complit,
        w.fio,
        w.id_employee,
        w.statusid,
        w.print_date,
        w.sendtoclientdate,
        w.sendtodoctordate,
        w.sendtoorgdate,
        w.sendtofax
    from works w
    where w.is_del <> true
    order by w.id_work desc
    limit 3000
)
```

### Зачем

В исходной функции список строился по всем заказам:

```sql
from Works
left outer join WorkStatus ...
where WORKS.IS_DEL <> 1
order by id_work desc
```

А клиент снаружи делал:

```sql
select top 3000 * from dbo.F_WORKS_LIST()
```

Проблема: функция могла сначала обработать все `50 000` заказов, а уже потом внешний запрос отрезал `3000`.

В оптимизированной версии мы сразу внутри функции берем:

```sql
limit 3000
```

work_ids as (

## 9. Добавлен CTE `workitem_counts`

### Проверка количества вызовов функций через track_functions

Чтобы подтвердить проблему не только расчетом, но и мониторингом, я включил сбор статистики по пользовательским функциям:

```sql
alter system set track_functions = 'all';
select pg_reload_conf();
```

Перед замером статистика была сброшена:

```sql
select pg_stat_reset();
```

Затем был выполнен исходный запрос:

```sql
select * from f_works_list() limit 3000;
```

После этого я проверил статистику вызовов функций:

```sql
select
    funcname,
    calls,
    round(total_time::numeric, 2) as total_time_ms,
    round(self_time::numeric, 2) as self_time_ms
from pg_stat_user_functions
where funcname in (
    'f_works_list',
    'f_workitems_count_by_id_work',
    'f_employee_fullname'
)
order by total_time desc;
```

Результат до оптимизации:

```text
funcname                       | calls | total_time_ms | self_time_ms
-------------------------------+-------+---------------+-------------
f_workitems_count_by_id_work   | 6000  | 2560.44       | 2560.44
f_employee_fullname            | 3000  | 184.31        | 184.31
f_works_list                   | 1     | 3028.19       | 283.44
```

По этим данным видно, что основная проблема действительно находилась в построчном вызове `f_workitems_count_by_id_work`: функция подсчета была вызвана `6000` раз для одного клиентского запроса.

После оптимизации статистика снова была сброшена:

```sql
select pg_stat_reset();
```

И выполнен тот же запрос:

```sql
select * from f_works_list() limit 3000;
```

Результат после оптимизации:

```text
funcname                       | calls | total_time_ms | self_time_ms
-------------------------------+-------+---------------+-------------
f_works_list                   | 1     | 67.02         | 21.18
f_workitems_count_by_id_work   | 1     | 28.74         | 28.74
f_employee_fullname            | 0     | 0.00          | 0.00
```

Сравнение количества вызовов:

```text
f_workitems_count_by_id_work:
до оптимизации    = 6000 вызовов
после оптимизации = 1 вызов
```

Именно это подтверждает главный смысл оптимизации: вместо построчной обработки был сделан наборный расчет.

### Добавилось

```sql
workitem_counts as (
    select c.*
    from work_ids ids
    cross join f_workitems_count_by_id_work(ids.ids) c
)
```

### Зачем

Вот здесь происходит главный выигрыш.

Раньше внутри `F_WORKS_LIST` было так:

```sql
dbo.F_WORKITEMS_COUNT_BY_ID_WORK(works.Id_Work,0) as WorkItemsNotComplit,
dbo.F_WORKITEMS_COUNT_BY_ID_WORK(works.Id_Work,1) as WorkItemsComplit
```

То есть два вызова функции на каждую строку заказа. Этот проблемный фрагмент есть в исходной функции.

Теперь стало:

```sql
cross join f_workitems_count_by_id_work(ids.ids) c
```

То есть функция вызывается **один раз** и возвращает таблицу счетчиков.

В файле это прямо описано: количество вызовов функции меняется с `3000 * 2 = 6000` на `1 вызов`.

## 10. В `SELECT` заменены два вызова функции на чтение готовых счетчиков

### Было

```sql id="xl366s"
dbo.F_WORKITEMS_COUNT_BY_ID_WORK(works.Id_Work,0) as WorkItemsNotComplit,
dbo.F_WORKITEMS_COUNT_BY_ID_WORK(works.Id_Work,1) as WorkItemsComplit,
```

### Стало

```sql id="u9pksn"
coalesce(wic.workitemsnotcomplit, 0) as workitemsnotcomplit,
coalesce(wic.workitemscomplit, 0) as workitemscomplit,
```

### Зачем

Теперь счетчики уже посчитаны в `workitem_counts`.

Поэтому в основном `select` мы просто берем готовые значения:

```sql id="w7fd47"
wic.workitemsnotcomplit
wic.workitemscomplit
```

А `coalesce(..., 0)` нужен на случай, если у заказа нет элементов `workitem`. Тогда после `left join` будет `NULL`, а по смыслу нужно вернуть `0`.

## 11. В `FROM` добавлен `left join workitem_counts`

### Добавилось

```sql id="t2kqiz"
left join workitem_counts wic
    on wic.id_work = tw.id_work
```

### Зачем

`workitem_counts` содержит строки вида:

```text id="0bayva"
id_work | workitemsnotcomplit | workitemscomplit
```

А `top_works` содержит сами заказы.

Чтобы вернуть заказ вместе с его счетчиками, мы соединяем их по:

```sql id="dppowi"
wic.id_work = tw.id_work
```

`LEFT JOIN` нужен, чтобы не потерять заказ, если у него нет элементов заказа.

## 12. Что в итоге оптимизировано

Ключевая проблема исходного кода была здесь:

```sql id="udgsxg"
F_WORKITEMS_COUNT_BY_ID_WORK(works.Id_Work,0)
F_WORKITEMS_COUNT_BY_ID_WORK(works.Id_Work,1)
```

В исходной функции `F_WORKS_LIST` эти вызовы стоят прямо в `SELECT`.

Если клиент просит 3000 заказов, это:

```text
3000 * 2 = 6000 вызовов функции подсчета
```

А если функция до внешнего `limit/top` обрабатывает все 50 000 заказов, то потенциально:

```text
50 000 * 2 = 100 000 вызовов
```

После оптимизации:

```text
1. Сначала выбираем 3000 заказов.
2. Собираем их id в массив.
3. Один раз вызываем функцию подсчета.
4. Внутри функции одним GROUP BY считаем оба счетчика.
5. Присоединяем результат обратно к заказам.
```

Главная замена:

```sql
f_workitems_count_by_id_work(id_work, false)
f_workitems_count_by_id_work(id_work, true)
```

на:

```sql
cross join f_workitems_count_by_id_work(ids.ids)
```

То есть вместо тысяч построчных вызовов — один наборный расчет.

## 5. Итог

В ходе работы были выполнены следующие действия:

1. Исходный MS SQL-скрипт был перенесен на PostgreSQL.
2. Был написан SQL-скрипт генерации тестовых данных:
   - 50 000 заказов;
   - 150 000 элементов заказа;
   - в среднем 3 элемента на заказ.
3. Были выявлены основные проблемы исходной функции:
   - построчные вызовы функции подсчета;
   - два вызова подсчета на каждый заказ;
   - повторные обращения к `workitem`;
   - построчный вызов функции ФИО;
   - потенциальная обработка всех 50 000 заказов до внешнего `limit 3000`;
   - форма `NOT IN`;
   - недостаточная пригодность имеющихся индексов для целевого сценария.
5. Была предложена оптимизация:
   - сначала выбрать 3000 заказов внутри `f_works_list`;
   - собрать их `id_work` в массив;
   - один раз вызвать `f_workitems_count_by_id_work(integer[])`;
   - внутри функции посчитать оба счетчика через `count(*) filter` и `group by`;
   - присоединить результат к заказам через `left join`.
6. Количество вызовов функции подсчета было уменьшено с тысяч вызовов до одного вызова.
